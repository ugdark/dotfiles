# 意図の書き分けガイド

「何をどこに書くか」を4つに分ける。ここで言うコメントは **実装（インライン）コメント**であって
API doc（JavaDoc/docstring 等）ではない。それぞれの場所にその役割だけを書き、重複させない。

| 置き場所 | 書くこと | 書かないこと |
|---|---|---|
| **コード** | **How**（どう実現するか） | 手順を追うだけのコメント（コード自体で語る） |
| **テストコード** | **What**（何を保証するか） | 実装の詳細（テスト名・アサーションで意図を表す） |
| **コミットログ** | **Why**（なぜこの変更をするか。背景・目的） | What（diffが語る。変更内容の逐条列挙をしない） |
| **実装コメント** | **Why not**（なぜ他の手段を採らないか。制約・罠・非自明な判断） | How（コードを読めば分かることを繰り返さない） |

## 適用の要点

- **コード = How**: 命名と構造で「どうやっているか」を読ませる。コメントで手順を追わない。
- **テスト = What**: テスト名で「何を保証するか」を宣言し、アサーションでそれを固定する。実装手順ではなく期待する振る舞いを書く。
- **コミット = Why**: 「なぜ必要か／どんな問題を解くか」を書く。変更内容そのもの（What）はdiffに任せる。
- **実装コメント = Why not**: 自明でない判断の理由をコード内に残す。「なぜこの方法か・なぜ別案を捨てたか・どんな罠・制約があるか」。Howの再説明はしない。

## publicメソッド

公開メソッドの実装ほど、後から読む人・呼ぶ人が判断の背景を知りたくなる。
非自明な選択や制約があれば、**実装コメントとして Why not を残す**（API doc の話ではない）。

- 「なぜこの実装にしたか・なぜ単純な別案では駄目か・呼び出し時の落とし穴」を優先して書く。
- 自明な処理にコメントを足さない。Why not が無い（自明な）実装は無理にコメントしない。

## プロジェクト固有の委任

- コメントスタイルや「どこまで残すか」の細かい基準は各プロジェクトの CLAUDE.md / コーディング規約に従う。
  本skillは言語・FWに依存しない原則のみを定める。

## 使い方

- コード・テスト・コミット・実装コメントを書く/レビューする際に、上表に沿って意図の置き場所を整える。
- `/d-guide-comment [対象]` で、既存コード（特に public メソッド）に Why not コメントが欠けていないかを点検し、補う提案をする。
