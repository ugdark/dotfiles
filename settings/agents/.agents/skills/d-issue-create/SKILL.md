# issue作成

チケット記述ルールに従い、リポジトリのissueテンプレートを尊重してGitHub issueを作成する。

## 使用場面

Claudeは以下の状況で**自動的にこのスキルを適用**する：

- 「issue作って」「issueを作成して」「チケット作って」と言われた時
- 会話中の課題・タスクを「issueにして」と言われた時

また、`/d-issue-create` で手動呼び出しも可能。

## チケット記述ルール（本文の品質基準）

issueは以下を満たすように書く（各ルールの詳しい根拠は knowledge-base の記事「チケットの書き方（完了条件とAC）」`scrum/チケットの書き方_完了条件とAC.md` を参照）：

1. **目的と完了条件をしっかり書く**: 「なぜやるか」と「何をもって終わりか」が本文だけで分かること
   - Why: 「完了」の定義は人によって異なる。明記は認識相違の防止であり、透明性（正しい情報が1箇所に集まり次の行動が誘発される状態）の最小単位
2. **完了条件（AC）は必ず1つ以上**: OK/NGが機械的に判定できる文にする。曖昧な条件（「いい感じにする」等）は不可
   - Why: AC＝受け入れ条件＝検証。検証できない条件はACとして成立しない。ACは解決策ではなく意思（何を満たすべきか）を述べる
3. **ACのフォーマットは2択**: 規則指向（条件の箇条書き）またはシナリオ指向（Given/When/Then）。振る舞いに着目する場合はシナリオ指向を選ぶ
   - Why: 「正しい」フォーマットは存在しないが、最初はテンプレートに寄せた方が安定した構造を提供できる
4. **スコープ外（やらないこと）を書く**: 期待の膨張を防ぐ。やってはいけないことがあれば必ず明記する
   - Why: やってはいけないことの記載はACのテクニックの1つ。スコープクリープと「これもやってくれると思った」を防ぐ
5. **8時間以内で終わる粒度にする**: 超えそうな内容は分割を提案する（親issue＋子issue、またはチェックリスト分割）
   - Why: 粒度を固定すると見積り不要になり、デイリーで進捗異常を機械的に検出できる
6. **短く書く**: 差分や実装から読み取れる詳細は書かない。背景・意図・判断基準だけを書く

**Doneとの区別**: チーム共通の完了定義（テスト通過・レビュー・デプロイ等のDone）は issue 個別には書かない。issueに書くのはそのissue固有のACのみ。

## 実行フロー

### Step 1: 材料の確認

会話の文脈と `$ARGUMENTS` から以下を抽出する：

- タイトル案
- 目的（なぜやるか）
- 完了条件の材料

**目的または完了条件が会話から読み取れない場合のみ**、AskUserQuestion で不足分を確認する（質問は1つずつ）。読み取れる場合は質問せず進む。

### Step 2: リポジトリのissueテンプレートを探す

Glob で以下を検索する：

1. `.github/ISSUE_TEMPLATE/*.md` / `.github/ISSUE_TEMPLATE/*.yml` — 複数あれば内容に合うものを選ぶ
2. `.github/issue_template.md` — 旧形式

**見つかった場合**: Read で読み込み、その構造に従う（チケット記述ルールはセクションの中身の書き方として適用）。
**見つからなかった場合**: 以下のデフォルト構造を使用する。

```markdown
## 目的
<なぜやるのか。背景と狙いを1〜3文>

## やること
<作業内容の箇条書き。3〜5項目まで>

## 完了条件
- [ ] <OK/NGが判定できる条件>

## スコープ外
- <このissueでやらないこと>
```

### Step 3: 本文を作成して粒度チェック

1. チケット記述ルールに従って本文を作成する
2. 作業量が8時間を超えそうな場合は、分割案を提示してユーザーに確認する

### Step 4: ユーザー確認

作成前に **タイトルと本文を表示してユーザーの承認を得る**（issueはPRと違いドラフトがないため、公開前に必ず確認する）。

### Step 5: issueを作成

```bash
gh issue create \
  --assignee @me \
  --title "タイトル" \
  --body "$(cat <<'EOF'
issue本文
EOF
)"
```

- ラベル指定がプロジェクトのCLAUDE.mdにあれば `--label` を付ける

**分割時（またはユーザーが「sub-issueで」と指定した時）**:

1. 親issueを先に作成する
2. 子issueを作成し、**ネイティブsub-issueとして親に紐付ける**（本文に親URLを書くだけの旧方式は使わない）
   - 優先: GitHub MCP の `sub_issue_write`（method: add）
   - フォールバック: `gh api graphql` の `addSubIssue` mutation

### Step 6: GitHub Projectへの追加

プロジェクトのCLAUDE.mdにGitHub Project指定（owner・project番号またはタイトル）がある場合、作成したissueをProjectに追加する：

```bash
gh project item-add <番号> --owner <owner> --url <issueのURL>
```

- 指定がなければこのStepはスキップする
- 分割時は親・子すべてを追加する

### Step 7: 完了報告

- 作成したissueのURLを表示（分割時は親子関係も表示）

## 重要ルール

- **テンプレートを尊重**: リポジトリにissueテンプレートがあれば必ずその構造に従う
- **作成前確認は必須**: Step 4を省略しない
- **完了条件なしのissueは作らない**: 材料が揃わないなら質問して埋める
- **CLAUDE.mdの指示を優先**: プロジェクトのCLAUDE.mdにlabel・assignee・issueタイプ・GitHub Project等の指定があればそれに従う
- **分割はネイティブsub-issueで**: 親子関係はsub-issue API（MCP `sub_issue_write` または GraphQL `addSubIssue`）で張る。本文へのURL記載だけで済ませない
- **個人設定ファイルへの参照を含めない**: issue本文に個人ローカルにしか存在しないパス・識別子（`~/.dotfiles/...`、`.desk/quests/...`、個人questの識別子等）を含めない。リポジトリ内にcommit済みのファイルやissue/チケットURLのみ参照する
