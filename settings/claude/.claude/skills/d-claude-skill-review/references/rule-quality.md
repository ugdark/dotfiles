# Rule品質基準

## 1. frontmatter検証

### 任意フィールド
- `paths`: Globパターンの配列。ルールの適用スコープを限定する
  - 例: `["**/test/**"]`, `["modules/**"]`, `["*.ts", "*.tsx"]`

### pathsチェックポイント
- 汎用ルール（全ファイル共通）なら paths は不要
- 特定ディレクトリ・ファイル種別に限定すべきルールに paths が設定されているか
- Globパターンが正しいか（`**` の使い方、拡張子指定）
- スコープが広すぎ/狭すぎないか

## 2. 本体コンテンツ品質

### 明確性
- ルールが曖昧さなく伝わるか
- 「なぜ」このルールが必要かが読み取れるか
- 具体例（コード例、Good/Bad比較）があるか
- 参照先（`@docs/...` や外部スキル）が有効か

### 構成
- 見出し（##）で論理的に分類されているか
- 関連するルールがグループ化されているか
- 1ファイルに詰め込みすぎていないか（分割の検討）

### 記述スタイル
- 簡潔で直接的な表現（「〜する」「〜を使用する」）
- 禁止事項は明確に記述（「〜しない」「〜を避ける」）
- 例外条件が明記されているか

## 3. スコープ設計

### 分割の判断基準
- 異なるpathsスコープが必要なルール → 別ファイルに分割
- 同じスコープ内の関連ルール → 1ファイルにまとめる
- ファイルが肥大化（目安: 100行超） → トピック別に分割を検討

### ファイル命名
- 内容が推測できる名前（`testing.md`, `naming-conventions.md`）
- kebab-case推奨
- `rules-` プレフィックスは冗長（ディレクトリで判別可能）

## 4. アンチパターン

- pathsなしで特定ファイル種別にしか適用できないルール
- 抽象的すぎて実行不能なルール（「良いコードを書く」等）
- CLAUDE.mdと重複する内容（rulesに書くべきはCLAUDE.mdで管理しきれないスコープ付きルール）
- 参照先（`@docs/...`）が存在しないパスを指している
- 1ファイルに無関係なルールが混在
- プロジェクト固有すぎてdotfilesに置けない内容（→ プロジェクトのrulesに置くべき）
