# Skill品質基準

## 1. frontmatter検証

### 必須フィールド
- `name`: kebab-caseの一意な識別子
- `description`: 50〜500文字、3人称で記述

### 任意フィールド
- `user-invocable`: 手動呼び出し可否
- `disable-model-invocation`: 自動発火の抑止
- `allowed-tools`: 使用ツールの制限
- `argument-hint`: 引数のヒント表示

### description品質チェック
- 3人称スタイルで書かれているか（「〜を行う」「〜に使用する」）
- 具体的なトリガーフレーズを含むか（「〜と言われた時」「〜の場合」）
- 短すぎないか（50文字未満はNG）、長すぎないか（500文字超はNG）
- 曖昧な表現を避けているか

## 2. 本体コンテンツ品質

### 文字数ガイドライン
- 推奨: 1,000〜3,000語（日本語の場合は相当する分量）
- 5,000語超: references/への分割を強く推奨

### 記述スタイル
- 命令形・不定詞形で記述（「〜する」「〜を読む」）
- 「〜してください」「もし〜なら」は避ける
- AI向けのオンボーディングガイドとして書く

### 構成
- 明確なセクション分け
- 論理的なフロー（Step 1, Step 2...）
- 具体的な指示（ツール名、ファイルパス、判定条件を明記）

## 3. Progressive Disclosure

### ディレクトリ構成の確認
```
skill-name/
├── SKILL.md          # 必須: 手順のみ
├── templates/        # テンプレートファイル
├── references/       # 詳細ドキュメント
├── scripts/          # 実行スクリプト
├── assets/           # 出力リソース
└── evals/            # テストケース
```

### チェックポイント
- SKILL.mdに詳細な仕様書を詰め込んでいないか
- references/に移すべき内容がSKILL.mdにないか
- テンプレートはtemplates/に分離されているか
- SKILL.mdからreferences/への参照（Read指示）が明確か

## 4. アンチパターン

- 曖昧なトリガー記述（何をもって発火するか不明）
- SKILL.mdの肥大化（references/に分離すべき内容が混在）
- descriptionが2人称（「あなたが〜」）
- 重要なトリガーフレーズの欠落
- 言語・フレームワーク固有のハードコード（汎用性の欠如）
- examples/references/が有効なのに未使用
